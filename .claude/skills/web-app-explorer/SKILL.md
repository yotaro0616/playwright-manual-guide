# Web App Explorer

Webアプリケーションを体系的に探索し、全画面・全機能を構造化された調査結果レポートにまとめるスキル。アプリ全体の探索にも、特定機能だけの探索にも対応する。

## このスキルが必要な理由

ドキュメント作成やQAの前に、Webアプリの画面を漏れなく把握する必要がある。手動では見落としが起きやすく、特に複数ロールがある場合は整理が困難。このスキルはその探索作業を構造化・再現可能にする。

## ユーザーから取得すべき情報

1. **アプリURL**（ログインページ）
2. **アカウント情報**（メールアドレス + パスワード）
3. **探索スコープ** — 以下のどちらかを明確にする
   - **全体探索**: ロール名の一覧（例: 「生徒」「マネージャー」「管理者」）
   - **機能探索**: 対象機能の名称・アクセス方法（例: 「ヘッダーのチャットボットアイコンから開くモーダル」）
4. **安全制約** — やってはいけない操作（例: 「ユーザー削除禁止」「本番データ投入禁止」）

スコープが不明な場合は必ず確認する。全体探索と機能探索ではプロセスが異なるため、ここを曖昧にしたまま進めない。

## 探索プロセス

### フェーズ1: プロジェクトセットアップ

探索を始める前に、成果物を格納するプロジェクト構造を整える。既にプロジェクトディレクトリが存在する場合はこのフェーズをスキップし、`exploration/findings.md` の作成だけ行う。

#### 1-1. ディレクトリ構造の作成

プロジェクトの目的に応じたディレクトリを作成する。探索だけが目的なら `exploration/` だけでよいが、後続のマニュアル作成・スクリーンショット撮影まで見据えている場合は、以下のような構造を一括で作成しておくと手戻りがない。

```
project-root/
├── CLAUDE.md              # プロジェクト定義
├── docs/                  # マニュアル本文
├── assets/images/         # スクリーンショット
└── exploration/
    └── findings.md        # 探索結果
```

#### 1-2. CLAUDE.mdの作成

プロジェクトルートに `CLAUDE.md` を作成し、以下を記載する。ユーザーから取得した情報をここに集約することで、後続のスキル（マニュアル執筆、スクリーンショット撮影等）が参照できるようになる。

- **プロジェクト概要** — 何のためのプロジェクトか
- **LMS/アプリ情報** — URL、アカウント情報
- **成果物** — 最終的に作るファイルの一覧
- **ディレクトリ構造** — 上記で作成した構造
- **進行フロー** — 探索→レビュー→執筆→撮影の流れ
- **安全制約** — やってはいけない操作
- **執筆ルール** — 文体、記法（後続工程で必要な場合）

#### 1-3. findings.mdの初期化

`exploration/findings.md` を以下の骨格で作成する。

**全体探索の場合:**
```markdown
# Webアプリ探索結果

## 探索ステータス
- [ ] ロール1
- [ ] ロール2

## ロール別メニュー構成
（探索後に記載）

## 機能詳細
（探索後に記載）

## クロスロール機能
（探索後に記載）

## 要確認事項
（探索後に記載）
```

**機能探索の場合:**
```markdown
# [機能名] 探索結果

## 探索ステータス
- [ ] [機能名]

## アクセス方法
（探索後に記載）

## 機能の構造
（探索後に記載）

## 操作フロー
（探索後に記載）

## 要確認事項
（探索後に記載）
```

### フェーズ2: 探索の実行

#### 全体探索の場合

各ロールについて以下の手順で探索する。

1. **ログイン**し、デフォルトのランディングページを記録する
2. **サイドバー/ナビゲーションを記録** — 全メニュー項目とURLパスを一覧化
3. **全ページを巡回** — スクリーンショットを撮り、ページ構造を読み取る
4. **主要機能を操作** — ドロップダウン、アコーディオン、タブを開いて隠れたコンテンツを発見
5. **モーダル・オーバーレイを探索** — 下記「モーダル/オーバーレイの探索」を参照
6. **ページの階層構造を把握** — 親子関係を理解する
7. **ログアウト**してから次のロールに切り替え

#### 機能探索の場合

対象機能に集中して以下の手順で探索する。

1. **ログイン**し、対象機能へのアクセス方法を確認・記録する
2. **機能のUI構造を把握** — 全要素をリストアップ（ボタン、入力欄、アイコン、リンク等）
3. **全状態を網羅** — 機能が取りうる全状態を探索する（詳細は下記参照）
4. **操作フローを確認** — 主要な操作の流れを実際に試し、入力→結果を記録する
5. **エッジケースを確認** — 空状態、エラー状態、境界条件を確認する

### モーダル/オーバーレイの探索

モーダルやオーバーレイはURLが変わらないため、ページ巡回だけでは見落とす。以下の手順で網羅する。

1. **開閉トリガーの特定** — どのボタン/アイコンからモーダルが開くかを記録する
2. **全状態の探索** — モーダルが取りうる表示状態を洗い出す
   - サイズ変更（通常/全画面/最小化）
   - サイドバーやパネルの展開/折りたたみ
   - タブや入力モードの切り替え
3. **モーダル内の全操作を試す** — ボタン、リンク、入力欄をすべて操作して動作を記録する
4. **モーダルの永続性を確認** — 閉じた後にデータが保持されるか、ページ遷移しても開けるかを確認する

### ラベルなしアイコンの特定

`title`や`aria-label`が設定されていないアイコンボタンに遭遇した場合、以下の順序で特定する。

1. **実際にクリックして動作を観察する** — 最も確実な方法。クリック前後のスナップショット/スクリーンショットを比較する
2. **周囲のコンテキストから推測する** — 隣接する要素やセクションの文脈からアイコンの役割を判断する
3. **ホバーしてツールチップを確認する** — ツールチップが表示される場合がある
4. **JavaScriptで調べる** — SVGのクラス名（例: `iconify--lucide`）やdata属性からアイコン名を特定する

特定できなかった場合は「❓ 不明」マークを付けて要確認事項に記載する。推測で断定しない。

### フェーズ3: クロスロール検証（全体探索の場合のみ）

全ロールの個別探索後、ロール間にまたがる機能を特定する。

- ロールAが作成したデータがロールBの画面にどう表示されるか
- 承認・レビューのワークフロー
- 通知・フィードバックの流れ

ロール間を往復して双方の見え方を検証する。

### フェーズ4: 確信度マーキング

全ての発見事項に確信度を付与する。

| マーク | 意味 |
|--------|------|
| `✅ 確認済` | 実際に操作して動作を確認した |
| `⚠️ 要確認` | 画面は見たが、動作の意図や仕様に自信がない |
| `❓ 不明` | 操作できなかった/理解できなかった |

⚠️ と ❓ の項目は「要確認事項」セクションに集約し、ユーザーに質問としてまとめる。推測で先に進まない。

## 安全ルール

- **基本は閲覧のみ** — ブラウズして観察する。データを変更しない
- **作成前に確認** — テストデータの投稿が必要な場合は、事前にユーザーに確認する
- **削除禁止** — 削除ボタン、アカウント無効化、破壊的操作は一切行わない
- **設定変更禁止** — 権限、ロール、システム設定を変更しない
- ユーザーが明示的に制約を緩和した場合（例:「テストデータの投稿OK」）はその指示に従う。ただし取り消し不可能な操作は必ず事前確認する。

## 出力フォーマット

### 全体探索の場合

最終的な `findings.md` には以下を含める。

1. **ロール別メニューテーブル** — メニュー名、URLパス、説明を各ロールごとに
2. **機能詳細** — 番号付きセクション、確信度マーク付きで各機能のUIと動作を記述
3. **クロスロール機能** — 複数ロールにまたがるワークフローを文書化
4. **要確認事項** — ユーザーへの具体的な質問を一覧化
5. **共通UI要素** — ヘッダー、サイドバー、通知など共有コンポーネント

### 機能探索の場合

最終的な `findings.md` には以下を含める。

1. **アクセス方法** — 機能への導線（どこから、どうやって開くか）
2. **UI構造** — 全要素の一覧と配置をテーブルで整理（要素名、位置、機能、確信度）
3. **状態一覧** — 機能が取りうる全状態を列挙（表示パターン、モード切替等）
4. **操作フロー** — 主要な操作の手順と結果を時系列で記述
5. **要確認事項** — ユーザーへの具体的な質問を一覧化

## 探索のコツ

- 全ページで最下部までスクロールする — 重要なコンテンツがフォールド下にあることが多い
- 全タブ・アコーディオン・ドロップダウンをクリックする — 初期表示だけで判断しない
- フィルター・ソート機能を確認する — 隠れたカテゴリやステータスが判明することがある
- 空の状態を記録する — 「データなし」メッセージからその機能が何を期待しているかがわかる
- URLパターンに注目する — アプリの情報アーキテクチャが読み取れる
- モーダルを開いたら全画面表示も試す — 通常サイズでは見切れている要素がある場合がある
- アイコンボタンは迷ったらクリックして確認する — ラベルがないアイコンの役割は実際の動作が最も確実な手がかり
