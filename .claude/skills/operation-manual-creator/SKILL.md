# Operation Manual Creator

Webアプリの機能情報から、ロール別の操作マニュアルとFAQを生成するスキル。

## このスキルが必要な理由

操作マニュアルは「何ができるか」ではなく「どう操作するか」をアクション単位で示す必要がある。このスキルはルールを標準化し、一貫した品質のマニュアルを生成する。

## ユーザーから取得すべき情報

1. **アプリの機能情報** — exploration/findings.md またはユーザーからの口頭説明
2. **ロール一覧** — 誰向けのマニュアルを作るか
3. **メディア方針** — 各ロールで画像/動画のどちらを使うか（デフォルト: 全ロール画像）
4. **出力形式** — Markdown / HTML / PDF など
5. **画像の公開方式** — 以下のどちらかを確認する
   - **ローカルパス（デフォルト）**: `assets/` 配下の相対パスで参照する
   - **S3公開**: AWS S3にアップロードして公開URLで参照する（Notionなど外部サービスに貼り付ける場合に必要）。S3バケットのベースURLをユーザーに確認する

## プロジェクト構造

以下のディレクトリ構造を作成する。メディア方針に応じて `images/` または `videos/` を使い分ける。

```
docs/
├── {role-1}/
│   ├── manual.md
│   └── faq.md（必要に応じて）
├── {role-2}/
│   └── manual.md
└── {role-3}/
    └── manual.md
assets/
├── {role-1}/images/   （画像方針のロール）
├── {role-2}/images/   （画像方針のロール）
└── {role-3}/videos/   （動画方針のロール）
```

## マニュアル構成ルール

### 全ロール共通

- **文体**: です/ます調
- **画面名・ボタン名**: 「」で囲む（例: 「ログイン」ボタン）
- **アクションベース**: 「〜するには」形式でセクションを構成
- **セクション番号は1始まり**（見出しに「Chapter」は付けない。`## 1. はじめに` の形式）

### セクション粒度のルール

1つの操作フローに属する関連アクションは、同じセクションにまとめる。読者の視点で「この操作をするためにこのページを開く」という単位でセクションを切る。

**やりがちな失敗（冗長な分割）:**
```
### 2-1. アイコンをクリックしてモーダルを開く
### 2-2. モーダルの画面構成を確認する
### 2-3. 全画面表示に切り替える
### 2-4. モーダルを閉じる
```

**正しい粒度:**
```
### 2-1. チャットボットを開く         ← 開く+全画面を1セクションに
### 2-2. 画面構成                     ← テーブルで要素一覧、閉じ方の補足もここ
```

判断基準:
- 同じ画面内の操作は同じセクションにまとめる
- 「開く→全画面にする→閉じる」のような表示制御は1セクションで十分
- 独立した操作フロー（「質問する」「過去質問を見る」等）は別セクションにする

### 画像方針（デフォルト）

手順の各ステップ直後に、そのステップに対応する画像1枚を配置する。

```markdown
### セクション名

1. ステップの説明文
![画像の説明](../../assets/{role}/images/filename.png)
<!-- 📸 撮影指示: 具体的な撮影内容 -->
2. 次のステップの説明文
![画像の説明](../../assets/{role}/images/filename.png)
<!-- 📸 撮影指示: 具体的な撮影内容 -->
```

**Notion互換性**: ステップと画像の間に空行を入れない。空行を入れるとNotionにコピペした際に番号付きリストが分断され、全ステップが「1.」にリセットされる。

**重要: ステップ統合ルール**
- アクション（クリック等）とその結果（画面表示）が同じ画面になる場合は、**1つのステップにまとめて画像1枚**にする
- 例: 「サイドバーの『設定』をクリックします」+「設定画面が表示されます」→ 「サイドバーの『設定』をクリックすると、設定画面が表示されます。」+ 画像1枚
- 全ステップに画像を配置する（例外なし）
- 1ステップ = 1画像（重複なし）

**画像ファイル名**: `セクション番号-ステップ番号-簡潔な英語名.png`（例: `4-1-1-content-list.png`）

**撮影指示の書き方**:
- どの画面（URL/ページ名）か
- どの状態（初期表示/入力済み/クリック後など）か
- 何が見えるべきか

### 動画方針（ユーザーが指定した場合のみ）

セクション単位で「概要文→動画→手順」の構成にする。

```markdown
### セクション名

セクションの概要説明文（1〜2文）

<!-- 🎬 動画挿入ポイント: 動画の内容説明 -->
<!-- 📹 撮影指示: 開始画面 → 操作手順 → 終了タイミング -->
<!-- ファイル名: ../../assets/{role}/videos/filename.mp4 -->

1. ステップ1の説明
2. ステップ2の説明
3. ステップ3の説明
```

**ステップ統合ルール（共通）**: アクション+結果が同じ画面なら1ステップに統合。

**動画ファイル名**: `セクション番号-簡潔な英語名.mp4`（例: `3-1-user-list.mp4`）

## セクション削除・リナンバー時のチェックリスト

セクションを削除する場合、以下を漏れなく対応する。1つでも漏れると参照切れや番号ズレが残る。

1. **マニュアル本文**: 該当セクションのテキスト・画像参照を削除
2. **セクション番号のリナンバー**: 後続セクションの番号を詰める（例: 4-6→4-5, 4-7→4-6）
3. **画像ファイルのリネーム**: セクション番号に対応するファイル名を変更（例: `4-7-1-*.png` → `4-6-1-*.png`）
4. **画像参照パスの更新**: マニュアル内の `![...](パス)` を新ファイル名に合わせる
5. **不要画像の削除**: 削除したセクションの画像ファイルを `assets/` から削除
6. **FAQ・他ドキュメントの参照更新**: FAQや他マニュアルで「→ 詳しくは 4-6 をご覧ください」等の参照番号を修正
7. **Chapter見出しのリナンバー**: Chapter自体を削除した場合、後続Chapterの番号も詰める

## FAQ作成ルール

- **Q&A形式**で記載
- **カテゴリ別**に整理（ログイン、教材、設定など）
- 回答は簡潔かつ具体的に
- 操作手順が必要な場合は番号付きリストを使用
- マニュアルの該当セクションへの参照を入れる（例: → 詳しくは「4-3. チュートリアルを選ぶ」をご覧ください）
- FAQにはメディア（画像/動画）は不要

## 進め方

1. **構成案（目次）を先に作成** → ユーザーレビュー
2. **承認後に本文を執筆** → メディアプレースホルダー（📸撮影指示）付き
3. **レビュー・修正** → ユーザーフィードバックに基づいて調整
4. **メディア挿入** → 撮影後に一括置換
