# Explore

Playwright を使って Web アプリを体系的に探索し、構造化された調査結果レポートを生成するスキル。アプリ全体の探索にも、特定機能だけの探索にも対応する。

## 前提

- CLAUDE.md にアプリ URL、アカウント情報、ロール一覧、安全制約が記載されていること
- CLAUDE.md に探索結果の出力先パスが記載されていること
- 上記がない場合は `/setup` の実行を案内して中断する

## 探索スコープの確認

実行時にユーザーの指示から探索スコープを判断する。

- **全体探索**: ロールごとに全画面を巡回する（デフォルト）
- **機能探索**: 特定の機能やページだけを探索する（ユーザーが対象を指定した場合）

スコープが不明な場合は必ず確認する。

## 探索プロセス

### 全体探索の場合

各ロールについて以下の手順で探索する。

1. **ログイン** — デフォルトのランディングページを記録する
2. **ナビゲーションを記録** — サイドバー/ヘッダーの全メニュー項目と URL パスを一覧化
3. **全ページを巡回** — ページ構造を読み取り、主要な UI 要素を記録する
4. **インタラクティブ要素を操作** — ドロップダウン、アコーディオン、タブを開いて隠れたコンテンツを発見
5. **モーダル・オーバーレイを探索** — 下記「モーダルの探索」を参照
6. **ページの階層構造を把握** — 親子関係、ページ間の遷移を理解する
7. **ログアウト** — 次のロールに切り替え

### 機能探索の場合

対象機能に集中して以下の手順で探索する。

1. **ログイン** — 対象機能へのアクセス方法を確認・記録する
2. **UI 構造を把握** — 全要素をリストアップ（ボタン、入力欄、アイコン、リンク等）
3. **全状態を網羅** — 機能が取りうる全状態を探索する
4. **操作フローを確認** — 主要な操作の流れを実際に試し、入力→結果を記録する
5. **エッジケースを確認** — 空状態、エラー状態、境界条件を確認する

### モーダルの探索

モーダルやオーバーレイは URL が変わらないため、ページ巡回だけでは見落とす。

1. **開閉トリガーの特定** — どのボタン/アイコンからモーダルが開くかを記録する
2. **全状態の探索** — サイズ変更、パネルの展開/折りたたみ、タブ切替等
3. **モーダル内の全操作を試す** — ボタン、リンク、入力欄をすべて操作して動作を記録する
4. **永続性を確認** — 閉じた後にデータが保持されるか確認する

### ラベルなしアイコンの特定

`title` や `aria-label` が設定されていないアイコンボタンは、以下の順序で特定する。

1. **クリックして動作を観察する** — 最も確実。前後のスナップショットを比較する
2. **周囲のコンテキストから推測する** — 隣接する要素やセクションの文脈から判断する
3. **ホバーしてツールチップを確認する**
4. **JavaScript で調べる** — SVG のクラス名や data 属性からアイコン名を特定する

特定できなかった場合は「❓ 不明」マークを付ける。推測で断定しない。

### クロスロール検証（全体探索の場合のみ）

全ロールの個別探索後、ロール間にまたがる機能を特定する。

- ロール A が作成したデータがロール B にどう表示されるか
- 承認・レビューのワークフロー
- 通知・フィードバックの流れ

### 確信度マーキング

全ての発見事項に確信度を付与する。

| マーク | 意味 |
|--------|------|
| `✅ 確認済` | 実際に操作して動作を確認した |
| `⚠️ 要確認` | 画面は見たが、仕様に自信がない |
| `❓ 不明` | 操作できなかった/理解できなかった |

⚠️ と ❓ の項目は「要確認事項」セクションに集約し、ユーザーへの質問としてまとめる。推測で先に進まない。

## 安全ルール

- **基本は閲覧のみ** — ブラウズして観察する。データを変更しない
- **作成前に確認** — テストデータの投稿が必要な場合は、事前にユーザーに確認する
- **削除禁止** — 削除ボタン、アカウント無効化、破壊的操作は一切行わない
- **設定変更禁止** — 権限、ロール、システム設定を変更しない
- CLAUDE.md の安全制約を遵守する
- ユーザーが明示的に制約を緩和した場合はその指示に従う。ただし取り消し不可能な操作は必ず事前確認する

## 出力

CLAUDE.md の「出力先」に記載されたパスに調査結果を保存する（デフォルト: `exploration/findings.md`）。

### 全体探索の出力内容

1. **ロール別メニューテーブル** — メニュー名、URL パス、説明を各ロールごとに
2. **機能詳細** — 番号付きセクション、確信度マーク付きで各機能の UI と動作を記述
3. **クロスロール機能** — 複数ロールにまたがるワークフローを文書化
4. **要確認事項** — ユーザーへの具体的な質問を一覧化
5. **共通 UI 要素** — ヘッダー、サイドバー、通知など共有コンポーネント

### 機能探索の出力内容

1. **アクセス方法** — 機能への導線
2. **UI 構造** — 全要素の一覧と配置
3. **状態一覧** — 機能が取りうる全状態
4. **操作フロー** — 主要な操作の手順と結果
5. **要確認事項** — ユーザーへの具体的な質問

## 探索のコツ

- 全ページで最下部までスクロールする — 重要なコンテンツがフォールド下にあることが多い
- 全タブ・アコーディオン・ドロップダウンをクリックする — 初期表示だけで判断しない
- フィルター・ソート機能を確認する — 隠れたカテゴリやステータスが判明することがある
- 空の状態を記録する — 「データなし」メッセージからその機能が何を期待しているかがわかる
- URL パターンに注目する — アプリの情報アーキテクチャが読み取れる
- モーダルを開いたら全画面表示も試す — 通常サイズでは見切れている要素がある場合がある
- アイコンボタンは迷ったらクリックして確認する — 実際の動作が最も確実な手がかり
